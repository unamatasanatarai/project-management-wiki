# Zakres i wymagania

Kontrola zakresu, wymagania, backlog, priorytety i kryteria akceptacji.

## Hasła w tej kategorii

| Hasło | Krótka definicja |
| --- | --- |
| [Scope](Haslo-Scope.md) | Zakres projektu określający wszystkie prace, funkcjonalności, deliverables i działania wymagane do osiągnięcia celów projektu. Scope definiuje granice projektu oraz stanowi podstawę planowania czasu, kosztów i zasobów. |
| [In Scope / Out of Scope](Haslo-In-Scope-Out-of-Scope.md) | Formalne określenie elementów objętych projektem oraz tych, które są wyłączone z realizacji. Pomaga zarządzać oczekiwaniami interesariuszy, ograniczać scope creep i minimalizować nieporozumienia dotyczące odpowiedzialności zespołu. |
| [Scope Creep](Haslo-Scope-Creep.md) | Niekontrolowane rozszerzanie zakresu projektu bez odpowiedniego zwiększenia budżetu, czasu lub zasobów. Scope creep jest jedną z głównych przyczyn opóźnień, przeciążenia zespołów i przekroczenia kosztów projektu. |
| [Requirements](Haslo-Requirements.md) | Wymagania definiujące potrzeby biznesowe, funkcjonalne i techniczne, które produkt lub system musi spełniać. Requirements stanowią podstawę projektowania, developmentu, testów i akceptacji rozwiązania. |
| [Functional Requirements](Haslo-Functional-Requirements.md) | Wymagania funkcjonalne opisujące konkretne funkcje i zachowania systemu, np. logowanie użytkownika, generowanie raportu lub obsługa płatności. Określają, co system ma robić z perspektywy użytkownika lub procesu biznesowego. |
| [NFR](Haslo-NFR.md) | Non-functional Requirements — wymagania niefunkcjonalne definiujące parametry jakościowe systemu, np. wydajność, bezpieczeństwo, skalowalność, dostępność, compliance lub usability. NFR wpływają na architekturę i stabilność rozwiązania. |
| [Business Requirements](Haslo-Business-Requirements.md) | Wymagania biznesowe definiujące cele organizacji, uzasadnienie projektu oraz oczekiwane korzyści biznesowe. Odpowiadają na pytanie, dlaczego projekt jest realizowany i jaką wartość ma dostarczyć. |
| [Technical Requirements](Haslo-Technical-Requirements.md) | Wymagania techniczne określające standardy, technologie, architekturę, integracje i ograniczenia infrastrukturalne niezbędne do realizacji rozwiązania. |
| [User Story](Haslo-User-Story.md) | Krótki opis potrzeby użytkownika zapisany z jego perspektywy, najczęściej w formacie: „Jako [rola] chcę [cel], aby [wartość]”. User Story koncentruje się na wartości biznesowej i stanowi podstawową jednostkę backlogu w Agile. |
| [Epic](Haslo-Epic.md) | Duży obszar funkcjonalny lub wysokopoziomowa inicjatywa obejmująca wiele powiązanych User Stories. Epic jest zbyt rozbudowany do realizacji w jednym sprincie i wymaga dekompozycji na mniejsze elementy. |
| [Feature](Haslo-Feature.md) | Funkcjonalność produktu dostarczająca konkretną wartość biznesową lub użytkową. Feature zazwyczaj obejmuje zestaw User Stories realizujących określony cel biznesowy. |
| [Acceptance Criteria](Haslo-Acceptance-Criteria.md) | Mierzalne warunki, które muszą zostać spełnione, aby User Story lub deliverable zostały zaakceptowane przez biznes, Product Ownera lub klienta. Acceptance Criteria stanowią podstawę testów i odbioru funkcjonalności. |
| [DoD](Haslo-DoD.md) | Definition of Done — wspólny dla zespołu zestaw kryteriów określających, kiedy zadanie uznaje się za ukończone. DoD może obejmować code review, testy, dokumentację, wdrożenie i spełnienie standardów jakości. |
| [DoR](Haslo-DoR.md) | Definition of Ready — zestaw warunków określających, kiedy backlog item jest wystarczająco przygotowany do rozpoczęcia prac. DoR minimalizuje ryzyko pracy nad niejasnymi lub niekompletnymi wymaganiami. |
| [Backlog](Haslo-Backlog.md) | Uporządkowana lista wszystkich zadań, wymagań, usprawnień i błędów związanych z projektem lub produktem. Backlog jest stale aktualizowany i priorytetyzowany zgodnie z wartością biznesową. |
| [Product Backlog](Haslo-Product-Backlog.md) | Centralna lista wszystkich funkcjonalności, usprawnień, zmian i bugów planowanych dla produktu. Product Backlog jest zarządzany i priorytetyzowany przez Product Ownera. |
| [Sprint Backlog](Haslo-Sprint-Backlog.md) | Zbiór backlog items wybranych przez zespół do realizacji w bieżącym sprincie wraz z planem wykonania prac potrzebnych do osiągnięcia Sprint Goal. |
| [Prioritization](Haslo-Prioritization.md) | Proces ustalania kolejności realizacji zadań na podstawie wartości biznesowej, ryzyka, zależności, kosztu i wpływu na cele produktu lub projektu. |
| [MoSCoW Prioritization](Haslo-MoSCoW-Prioritization.md) | Technika priorytetyzacji dzieląca wymagania na Must-have, Should-have, Could-have i Won’t-have. Pomaga definiować MVP oraz podejmować decyzje o zakresie projektu przy ograniczonych zasobach. |
| [Refinement / Grooming](Haslo-Refinement-Grooming.md) | Regularny proces analizy, doprecyzowywania, dekompozycji i estymacji backlog items w celu przygotowania ich do realizacji w kolejnych sprintach. |
| [Change Control](Haslo-Change-Control.md) | Formalny proces zarządzania zmianami w projekcie obejmujący ocenę wpływu zmian na zakres, budżet, harmonogram, ryzyka i zasoby przed ich zatwierdzeniem. |
| [Traceability Matrix](Haslo-Traceability-Matrix.md) | Macierz powiązań między wymaganiami, implementacją, testami i deliverables. Umożliwia śledzenie realizacji wymagań oraz analizę wpływu zmian na system. |
| [Use Case](Haslo-Use-Case.md) | Szczegółowy opis interakcji użytkownika lub systemu z aplikacją prowadzącej do osiągnięcia konkretnego celu biznesowego. Use Case definiuje scenariusze działania, kroki i wyjątki procesu. |
| [Persona](Haslo-Persona.md) | Fikcyjny profil reprezentujący typowego użytkownika produktu wraz z jego potrzebami, celami, problemami i zachowaniami. Persony wspierają projektowanie UX oraz podejmowanie decyzji produktowych. |
| [MVP](Haslo-MVP.md) | Minimum Viable Product — minimalna wersja produktu zawierająca kluczowe funkcjonalności potrzebne do dostarczenia wartości użytkownikowi i zebrania feedbacku rynkowego przy minimalnym koszcie inwestycji. |
